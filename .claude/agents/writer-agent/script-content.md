# 강의교안 콘텐츠 작성 (Phase 6) 세부 워크플로우

### 설계 원칙

| 차원 | 구성안 Phase 6 | 교안 콘텐츠 Phase 6 |
|------|--------------|-------------------|
| 산출물 성격 | 설계 문서 (What) | 콘텐츠 문서 (How) — 강사 발화문 없이 설명문 중심 |
| §4/§5 상세도 | 교시당 10~15줄 (활동 요약) | 교시당 80~120줄 (설명문+예시+활동+시각화) |
| 고유 요소 | 없음 | 설명문(문어체), 다중 예시(★/◇), You Do 완성 코드, 오개념 교정 테이블 |
| 발화문 포함 | - | 없음 (Phase 8 대본에서 생성) |
| 예상 분량 | 1,678행 (3일) | 3,500~5,000행 (3일) |
| 분할 모드 | 3-Part | Day별 분할 (days+2 Part) |
| GAIDE 매핑 | 동일 | 동일 (Setup→Draft→Macro→Micro→Integration) |

**5대 핵심 원칙**:

1. **GAIDE 5단계 적용**: 구성안과 동일
2. **콘텐츠 전용**: 설명문(문어체)으로 작성. 발화문(`> "..."`)은 Phase 8 대본에서 생성하므로 이 단계에서 포함하지 않는다
3. **다중 예시**: 각 핵심 개념에 최소 2~3가지 예시 제공. ★ 핵심(필수) + ◇ 심화(선택) 마커로 구분
4. **입력 충실성**: 5개 입력 파일의 데이터만 사용, 팩트 창작 금지
5. **You Do 완전성**: 시작 코드 + 단계별 가이드 + 완성 코드 + 예상 결과 + 흔한 실수 시나리오

**교안 콘텐츠 5대 원칙** (기술 교육 시 특히 엄격 적용):

| 원칙 | 교안 적용 |
|------|----------|
| 완전성 | 모든 코드·명령어는 복사 가능한 fenced code block(` ``` `)으로 제공. `[코드 시연: ...]` 같은 플레이스홀더 금지 |
| 명확성 | 코드 내 파일명·경로·클래스명을 정확히 표기. 코드 블록 상단에 파일명 주석 포함 |
| 재현성 | 각 코드 실행 후 예상 결과(콘솔 출력, API 응답, UI 상태)를 명시. 학습자가 혼자서 100% 재현 가능 |
| 추적성 | 코드 출처(context7 라이브러리명, 공식 문서 URL, brainstorm §4 소재 ID) 명시 |
| 비생략 | 코드를 임의로 요약하거나 `// ...` 로 생략하지 않음. 화면에 보여줄 코드 전체를 담음 |

### 금지 패턴 (산출물에 포함 시 GATE-6 FAIL)

다음 문구가 산출물에 포함되면 오케스트레이터의 bash 검증 스크립트(`gate-check-session.sh`)에 의해 자동 FAIL 처리된다. FAIL 시 해당 교시를 재작성해야 한다.

| 금지 문구 | 위치 | 대체 방법 |
|---------|------|---------|
| `구성안 비유 체계 참조.` | I Do > 비유/시각화 | brainstorm §4의 비유를 **5줄 이상 설명문으로 전개** |
| `선수 확인 질문` | 도입 > 형성평가 | **구체적 질문 내용과 예상 답변**을 작성 |
| `강사 시연을 따라 코드 작성` | We Do > 단계별 가이드 | **실제 단계별 코드 가이드** (스캐폴드 코드 포함) |
| `다음 차시 주제와의 연결점을 안내합니다.` | 정리 > 다음 차시 연결 | **구체적 연결 내용** ("오늘 배운 X가 다음 시간 Y의 기초가 됩니다") |
| `해당 SLO 달성 여부 확인` | 정리 > 성공 기준 | **구체적 성공 기준** ("Student가 ~를 독립적으로 수행할 수 있다") |
| 모든 차시 동일 `필요 자료` | 필요 자료 | 차시별 **실제 필요한 자료** 작성 (슬라이드 번호, 코드 파일명 등) |
| 모든 차시 동일 `흔한 실수` | You Do > 흔한 실수 | 차시 주제에 맞는 **고유한 실수 시나리오** 작성 |

### 분할 작업 모드 (Part Mode) — Bottom-Up 3계층

**분할 판단 기준**: `total_sessions` (총 교시 수)

| 조건 | 모드 | Part 수 |
|------|------|---------|
| `total_sessions ≤ 10` | 단일 모드 | 1 |
| `total_sessions > 10` | 세션별 분할 | sessions + 2 |

오케스트레이터가 architecture.md 시간표를 파싱하여 세션 목록을 동적으로 결정한다.

#### 세션별 분할 모드 — 1교시 단위 호출

| Part | 범위 | Step | 산출물 |
|------|------|------|--------|
| Part 0/{N} | §1~§3 | Step 0~2 | `_header.md` (Write 신규) |
| Part 1/{N} | §4 세션 D{day}-{num} | Step 3 (1교시) | `session_D{day}-{num}.md` (1파일 Write 신규) |
| ... | §4 세션 D{day}-{num} | Step 3 (1교시) | `session_D{day}-{num}.md` (1파일 Write 신규) |
| Part {N}/{N} | §5~§8 | Step 4~5 | `_footer.md` (Write 신규) |

**핵심**: 호출 단위는 **1교시**, 산출물은 **차시별 독립 파일**.

```
변경:  writer-agent(D1-1) → session_D1-1.md (1파일 Write)
```

이유: 콘텐츠 생성 집중도 확보 + 오케스트레이터가 해당 교시 입력을 발췌 전달

**예시: 3일×8시간 (24세션 → 26 Part)**

| Part | 범위 | 산출물 |
|------|------|--------|
| Part 0/26 | §1~§3 | `_header.md` |
| Part 1/26 | §4 세션 D1-1 | `session_D1-1.md` |
| Part 2/26 | §4 세션 D1-2 | `session_D1-2.md` |
| Part 3/26 | §4 세션 D1-3 | `session_D1-3.md` |
| ... | ... | ... |
| Part 24/26 | §4 세션 D3-8 | `session_D3-8.md` |
| Part 25/26 | §5~§8 | `_footer.md` |

**예시: 3일×4시간 (12세션 → 14 Part)**

| Part | 범위 | 산출물 |
|------|------|--------|
| Part 0/14 | §1~§3 | `_header.md` |
| Part 1/14 | §4 세션 D1-1 | `session_D1-1.md` |
| Part 2/14 | §4 세션 D1-2 | `session_D1-2.md` |
| ... | ... | ... |
| Part 12/14 | §4 세션 D3-4 | `session_D3-4.md` |
| Part 13/14 | §5~§8 | `_footer.md` |

#### 차시별 독립 파일 작성 규칙

각 `session_D{day}-{num}.md`에는 해당 교시의 **완전한 교안 콘텐츠**가 포함된다:

```markdown
# D{day}-{num}: {주제} ({분}분)

## 메타데이터
| 항목 | 내용 |
|------|------|
| SLO | {SLO 코드}: {목표} |
| content_type | {hands-on / concept / activity} |

## 도입 ({분}분)
**선수지식**: ...
**학습목표**: ...
**Hook/동기유발**: ...
**형성평가 (도입)**: ...

## I Do: 시범/설명 ({분}분)
### 핵심 개념
{설명문 — 문어체}
### 비유/시각화
{비유 전개 + Mermaid}
### ★ 예시 1: {제목} (핵심)
{코드 + 설명 + 예상 결과 + 핵심 교훈}
### ★ 예시 2: {제목} (핵심)
{코드 + 설명}
### ◇ 예시 3: {제목} (심화)
{코드 + 설명}
### 오개념 교정
{테이블: 오개념 | 올바른 이해 | 교정 방법}

## We Do: 안내 실습 ({분}분)
### 스캐폴드 코드
{빈칸/TODO 코드}
### 단계별 가이드
### 완성 코드
### 형성평가 (전개)

## You Do: 독립/그룹 실습 ({분}분)
### 실습 과제
### 시작 코드
### 단계별 가이드
### ★ 완성 코드 (정답)
### 예상 실행 결과
### 흔한 실수 시나리오
{테이블: 실수 | 에러 메시지 | 해결}

## 정리 ({분}분)
### 핵심 요약
### 실무 적용
### 다음 차시 연결
### 형성평가 (Exit)

## 필요 자료
```

- 각 session 파일은 **독립 실행 가능**: 해당 차시만으로도 학습 진행 가능
- 파일 시작에 Day 헤딩(`### Day {N}: {테마}`)은 포함하지 않는다 — 블록 병합 시 추가

#### 차시별 최소 분량 기준 (GATE-6 검증)

오케스트레이터의 `gate-check-session.sh`가 줄 수를 검증한다. 미달 시 해당 교시를 재작성한다.

| content_type | 최소 줄 수 | 근거 |
|-------------|----------|------|
| hands-on | ≥ 150줄 | I Do 완성코드 + We Do 스캐폴드 + You Do 시작/완성코드 포함 |
| concept | ≥ 120줄 | I Do 설명문 + 비유 전개 + ★ 예시 2개 포함 |
| activity | ≥ 100줄 | 활동 지시문 + 성공 기준 + 흔한 실수 포함 |

#### Overlap 컨텍스트 수신 규칙

블록 호출 시 이전 블록의 마지막 차시 내용을 참조하여 차시 간 전환의 일관성을 보장한다.

| 블록 순서 | Overlap 입력 |
|-----------|-------------|
| 첫 번째 블록 | 없음 |
| 두 번째 이후 | 이전 블록 마지막 차시 `session_*.md`의 **정리 섹션** (마지막 50~100줄) |

**Overlap 사용 방법**:
- 도입부 선수지식 항목에 이전 차시 핵심 내용 반영
- 다음 차시 연결 항목 작성 시 이전 차시의 정리 내용을 참조
- Overlap이 추출 실패하거나 빈 경우: 일반적인 연결 내용으로 작성

#### `_header.md` 포함 범위

Part 0에서 Write하는 `_header.md`에는 다음이 포함된다:
- §1 강의 개요 (기본 정보, 교수 모델)
- §2 학습 목표 요약 (CLO-SLO 매핑, 차시-SLO 배정 매트릭스)
- §3 공통 교안 구조 (교수 모델별 50분 내부 구조, GRR 패턴)
- §4 Day 헤딩 목록: 각 Day의 `### Day {N}: {테마}` 헤딩만 미리 작성 (교시 내용은 session 파일에)

#### `_footer.md` 포함 범위

Part {N}에서 Write하는 `_footer.md`에는 다음이 포함된다:
- §5 형성평가 집계 (차시별 형성평가 일람, 총괄평가 설계, 루브릭)
- §7 교재 및 참고자료
- §8 강사 가이드 (오개념 대응 카드, 시간 초과 축소 전략, 메타포, 트러블슈팅, 사전 확인)

`_footer.md` 작성 시 모든 `session_*.md` 파일을 Read하여 §4 인라인 내용에서 §5를 집계한다.

#### 분할 모드 공통 규칙

- 각 Part는 다른 Part가 작성한 파일을 수정하지 않는다
- 각 `session_*.md`는 완전한 단일 교시 교안 콘텐츠를 담는 독립 파일이다
- 각 Part는 메타데이터 헤더(차시, 교수모델, SLO)를 Part 입력 컨텍스트로 제공받는다
- 오케스트레이터가 prompt에 `블록 세션` 목록을 전달하므로, 해당 세션만 작성한다
- **통합(병합)은 writer-agent가 수행하지 않는다** — Phase 8 오케스트레이터가 담당

#### revision 모드 (차시 재작성)

Phase 7 블록별 검토에서 REVISION_REQUIRED 판정 시, 해당 블록의 차시 파일을 재작성하는 모드.

| 항목 | 내용 |
|------|------|
| 입력 | `session_D{day}-{num}.md` (해당 블록의 차시 파일들), `_review_content_{block_id}.md` (위반 목록 + 수정 가이드) |
| 도구 | Read, Write |
| 산출물 | `session_D{day}-{num}.md` (해당 차시 파일 전체 Write로 교체) |

**동작**:
1. `_review_content_{block_id}.md` Read → Major 위반 사항 + 수정 가이드 추출
2. 해당 블록의 `session_*.md` 파일들 Read → 현재 내용 파악
3. 위반 사항이 있는 차시만 재작성 (다른 차시 파일은 **절대 수정하지 않는다**)
4. Write 도구로 해당 `session_*.md` 파일 전체를 교체 (Edit가 아닌 **Write**)
5. 재작성 시에도 GRR 순서, 시간 합산 준수

#### 3-3a. 기술 교육 코드·시각화 규칙

**(A) 코드 배치 원칙** (content_type == "hands-on" 차시 I Do·We Do·You Do 공통, concept 차시 선택적):

| 코드 규모 | 배치 방법 |
|-----------|----------|
| 짧은 코드 (≤20줄) | 해당 섹션에 직접 전체 코드 블록 배치 |
| 중간 코드 (21~50줄) | 해당 섹션에 직접 전체 코드 블록 배치 + §7 코드 모음에도 포함 |
| 긴 코드 (>50줄) | 핵심 부분만 발췌하여 해당 섹션에 배치 + `code_examples_D{day}-{num}.md` 별도 파일에 전체 코드 작성 후 참조 링크 삽입 |

**(B) GRR 구간별 코드 형식**:

| GRR 구간 | 코드 형식 | 예시 |
|---------|---------|------|
| I Do | **완성 코드** — 시연할 전체 코드. 주석으로 핵심 포인트 표시 | `// ← @Service 어노테이션으로 Bean 등록` |
| We Do | **스캐폴드 코드** — 빈 칸/TODO가 있는 코드. 학습자가 채울 부분 표시 | `@___Mapping("/{id}") // TODO: HTTP 메서드` |
| You Do | **시작 코드 골격 + 완성 코드 + 예상 출력** | 시작 코드 → 완성 코드 → 예상 API 응답 JSON |

**(C) 코드 블록 작성 규칙**:
- 언어 지정 필수: ` ```java `, ` ```python `, ` ```bash ` 등
- 파일명 주석: 첫 줄에 `// 파일: src/main/java/com/example/CustomerService.java`
- 실행 결과: 코드 다음에 `**예상 결과**:` + 출력 코드 블록

**(D) 시각화 규칙** (아키텍처·흐름 설명 시):
- 아키텍처/데이터 흐름 → Mermaid `flowchart` 또는 `graph`
- 시간 순서 상호작용 → Mermaid `sequence diagram`
- 한글 레이블 → 따옴표 필수: `["한글 노드명"]`
- Mermaid 블록은 설명문 바로 아래에 배치

**(E) `[코드 시연: ...]` 플레이스홀더 금지 규칙** (content_type == "hands-on" 차시 적용):
- `[코드 시연: ...]`, `[코드 예시]`, `[데모]` 같은 행동 지시로 실제 코드를 대체하는 것을 **금지**한다
- 대신: 해당 섹션에 **실제 코드 블록**을 직접 작성
- 추가 금지: `구성안 비유 체계 참조.` — 비유/시각화 섹션에 참조 문구만 남기는 것을 금지. brainstorm §4의 비유를 설명문으로 전개해야 한다
