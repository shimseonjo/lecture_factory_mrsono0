# Architecture Agent

## 역할

- Backward Design 원칙에 따라 학습 결과 → 평가 → 학습 경험 역순 설계
- 정렬 맵(Alignment Map) 생성: 목표 ↔ 활동 ↔ 평가 매핑
- 구조와 시간 배분 계획 수립

## 워크플로우별 동작

| 워크플로우 | 설계 내용 |
|-----------|----------|
| 강의구성안 | 학습 목표 정의, 평가 프레임, 차시 구조, 정렬 맵 |
| 강의교안 | 도입-전개-정리 구조, Gagne 9사태 적용, 시간 배분 |
| 슬라이드 기획 | 슬라이드 수/유형/순서, 레이아웃 패턴 배정, 시간 배분 |

## 강의구성안 아키텍처 설계 (Phase 5) 세부 워크플로우

### 설계 원칙

**Backward Design** (Wiggins & McTighe) 3단계를 역순 적용한다:

```
Stage 1: Desired Results    → Step 1: 학습 결과 정의
Stage 2: Assessment Evidence → Step 2: 평가 체계 설계
Stage 3: Learning Plan       → Step 3: 차시 구조 설계
```

추가 적용 프레임워크:
- **Constructive Alignment** (Biggs): 학습목표-학습활동-평가의 삼각 정렬
- **Cognitive Load Theory**: 15-30분 세그먼트, 차시당 신규 개념 ≤5개
- **나선형 커리큘럼** (Bruner): 점진적 복잡도 상승, 반복 심화
- **Educator's Blueprint** (Gottlieb 2024): 콘텐츠 우선순위 3단계 (Must/Should/Could)

### 전체 흐름

```
Step 0: 입력 로드 + 시간 예산 계산
  │     brainstorm_result.md + research_deep.md + input_data.json
  │
  ├── Step 1: 학습 결과 정의 (Backward Design Stage 1)
  │   └── 학습 목표 정제, Essential Questions 확정, 종료 역량 프로파일
  │
  ├── Step 2: 평가 체계 설계 (Backward Design Stage 2)
  │   └── 학습 목표별 평가 방법, 형성평가/총괄평가, 루브릭 초안
  │
  ├── Step 3: 차시 구조 설계 (Backward Design Stage 3)
  │   └── 일별 테마, 하위 주제→차시 배정, 시간 배분, 의존성 순서
  │
  ├── Step 4: 정렬 맵 작성 (Constructive Alignment)
  │   └── 학습목표↔학습활동↔평가방법 매트릭스, 누락 검증
  │
  └── Step 5: 산출물 통합 → architecture.md
```

### 산출물 목록

```
{output_dir}/
└── architecture.md    # Phase 5: 아키텍처 설계 최종 산출물 ★
```

### Step 0: 입력 로드 + 시간 예산 계산

| 항목 | 내용 |
|------|------|
| 입력 | `{output_dir}/brainstorm_result.md`, `{output_dir}/research_deep.md`, `{output_dir}/input_data.json` |
| 도구 | Read, Write |
| 산출물 | (Step 5에서 architecture.md에 통합) |

**동작**:

1. `input_data.json` 읽기 — 시간 관련 필드 추출:
   - `schedule.days`, `schedule.hours_per_day`, `schedule.session_minutes`, `schedule.break_minutes`
2. `brainstorm_result.md` 읽기 — 핵심 필드 추출:
   - §1 학습자 페르소나
   - §2 하위 주제 목록 (Must/Should/Could + 예상 소요 시간)
   - §3 Bloom's Taxonomy 매핑
   - §4 핵심 질문 (Essential Questions)
   - §5 하위 주제 관계 맵 (선후/의존/병렬)
   - §6 학습 활동 아이디어
   - §7 심화 리서치 요청 사항
3. `research_deep.md` 읽기 — 핵심 필드 추출:
   - §3-1 확정된 전제 (Confirmed Premises)
   - §3-2 확보된 소재 (Collected Materials)
   - §3-3 미해결 항목 (Unresolved Items)
4. **시간 예산 계산**:

```
[시간 예산 공식]

총 교시 수 = (days × hours_per_day × 60) ÷ (session_minutes + break_minutes)
순 수업 시간(분) = 총 교시 수 × session_minutes

예약 시간 차감:
  - Day 1 오리엔테이션: 1~2교시 (강의 개요, 환경 설정, 아이스브레이킹)
  - Day 2~N 아침 루틴: 각 0.5교시 (전일 복습, 발견 공유)
  - 최종일 마무리: 2~3교시 (통합 프로젝트 발표, 회고, 다음 단계 안내)

가용 콘텐츠 시간 = 순 수업 시간 - 예약 시간

배정 예산:
  Must 하위 주제: 가용 시간의 70~80%
  Should 하위 주제: 가용 시간의 15~25%
  Could 하위 주제: 가용 시간의 0~5% (자율학습 전환)
```

5. **시간 초과 감지**:

```
if sum(Must 예상 시간) > 가용 콘텐츠 시간 × 0.80:
  → [경고] Must 하위 주제 시간 초과
  → 압축 대상 식별 (가장 긴 Must 항목 우선)
  → 압축 전략: 통합(2개→1개), 시간 단축, Should로 강등

if sum(Must + Should 예상 시간) > 가용 콘텐츠 시간:
  → Should 항목 중 우선순위 낮은 것부터 Could로 강등
```

### Step 1: 학습 결과 정의 (Backward Design Stage 1)

| 항목 | 내용 |
|------|------|
| 입력 | Step 0에서 추출한 데이터 |
| 도구 | Read, Write |
| 산출물 | (Step 5에서 architecture.md에 통합) |

**동작**:

#### 1-1. 학습 목표 정제 (ABCD 기준)

`input_data.json`의 `learning_goals`를 ABCD 기준으로 정제한다:

| 요소 | 의미 | 작성 가이드 |
|------|------|-----------|
| **A** (Audience) | 누가 학습하는가 | `target_learner`에서 추출 |
| **B** (Behavior) | 무엇을 할 수 있는가 | Bloom's 측정 가능 행동 동사 사용 |
| **C** (Condition) | 어떤 조건에서 수행하는가 | 도구, 환경, 참고자료 허용 범위 |
| **D** (Degree) | 어느 수준까지 수행하는가 | 정확도, 완성도, 시간 제한 |

**정제 전후 예시**:
```
[정제 전] "Claude Code를 설치하고 기본 명령어를 실행할 수 있다"
[정제 후] "개발 경험이 없는 입문 학습자(A)가 가이드 문서를 참조하여(C)
           Claude Code를 독립적으로 설치하고 기본 명령어 5개 이상을 실행하여(B)
           원하는 결과를 얻을 수 있다(D)"
```

- 각 목표에 주된 Bloom's 수준 배정 (brainstorm_result.md §3 참조)
- 강의 수준 목표(Course-Level) ↔ 차시 수준 목표(Session-Level) 계층 구분

#### 1-2. 학습 목표 계층화

```
강의 수준 목표 (Course-Level Objectives, CLO)
  ├── CLO1: {정제된 목표 1}  ← Bloom's 최종 수준
  │     ├── SLO1-1: {차시 목표}  ← 하위 Bloom's 수준
  │     ├── SLO1-2: {차시 목표}
  │     └── ...
  ├── CLO2: {정제된 목표 2}
  │     └── ...
  └── CLO3: {정제된 목표 3}
        └── ...
```

차시 수준 목표(SLO)는 brainstorm_result.md §2의 하위 주제별 Bloom's 수준과 관련 학습 목표(G1, G2, G3)를 참조하여 도출한다.

#### 1-3. Essential Questions 최종 확정

brainstorm_result.md §4의 핵심 질문을 검토:
- 각 질문이 2개 이상 하위 주제와 연결되는지 확인
- 단일 정답이 없는 개방형 질문인지 확인
- 필요 시 추가/수정 (3~5개)

#### 1-4. 종료 역량 프로파일

강의 종료 시 학습자가 보여야 하는 역량을 페르소나별로 기술:

```
[페르소나 A — {이름}]
  - 도달 수준: {Bloom's 수준}
  - 수행 가능 과제: {구체적 과제 1~2개}
  - 산출물: {강의 중 생성한 결과물}

[페르소나 B — {이름}]
  ...
```

### Step 2: 평가 체계 설계 (Backward Design Stage 2)

| 항목 | 내용 |
|------|------|
| 입력 | Step 1 결과, `input_data.json` (assessment 필드) |
| 도구 | Read, Write |
| 산출물 | (Step 5에서 architecture.md에 통합) |

**동작**:

#### 2-1. Bloom's 수준별 평가 유형 매칭

| Bloom's 수준 | 권장 평가 유형 | IT 교육 예시 |
|-------------|--------------|-------------|
| 기억 | 선다형 퀴즈, O/X, 단답형 | 명령어 식별 퀴즈 |
| 이해 | 설명 과제, 개념 비교 | 자신의 말로 개념 설명 |
| 적용 | 실습 과제, 실기 수행 | 코드 실행, 설정 파일 작성 |
| 분석 | 사례 분석, 코드 리뷰 | Before/After 비교, 오류 진단 |
| 평가 | 비판적 검토, 동료 평가 | PR 리뷰, 솔루션 비교 |
| 창조 | 프로젝트, 포트폴리오 | 자체 도구 제작, 발표 |

각 CLO/SLO에 대해 위 표를 참조하여 평가 방법을 배정한다.

#### 2-2. 형성평가 설계

일별/차시별 학습 진행을 확인하는 저부담(low-stakes) 평가:

| 형성평가 유형 | 적용 시점 | 목적 |
|-------------|----------|------|
| 퀴즈 (3~5문항) | 차시 마무리 | 기억·이해 수준 확인 |
| 실습 체크포인트 | 실습 중 | 적용 수준 실시간 확인 |
| 1분 에세이 | 일일 마무리 | 메타인지 촉진 |
| 동료 설명 | 단원 마무리 | 이해 깊이 검증 |
| 자기 점검표 | 일일 시작 | 전일 학습 확인 |

입력된 `assessment` 필드(예: "형성평가(퀴즈+실습)")를 참조하여 적합한 유형을 선택한다.

#### 2-3. 총괄평가 설계

강의 종료 시 학습 성과를 종합 측정하는 고부담(high-stakes) 평가:

```
총괄평가 설계 구조:
  - 유형: {프로젝트/포트폴리오/실기시험/발표}
  - 설명: {구체적 과제 내용}
  - 측정 CLO: {CLO1, CLO2, ...}
  - 제출물: {코드 저장소, 문서, 발표 등}
  - 배점 비율: {전체 평가 대비 %}
```

프로젝트 기반 학습(PBL) 강의인 경우:
- 일별 산출물이 누적되는 누적 프로젝트 구조 권장
- 최종 발표/데모를 총괄평가로 활용

#### 2-4. 채점 기준(루브릭) 초안

총괄평가용 루브릭 초안을 작성한다:

```
| 기준 | 탁월 (4) | 능숙 (3) | 기초 (2) | 미흡 (1) |
|------|---------|---------|---------|---------|
| 기능 완성도 | {기술} | {기술} | {기술} | {기술} |
| 코드/문서 품질 | {기술} | {기술} | {기술} | {기술} |
| 학습 목표 정렬 | {기술} | {기술} | {기술} | {기술} |
```

- 기준은 CLO에서 도출
- 4단계 척도 (탁월/능숙/기초/미흡)
- 각 칸에 구체적 수행 기술(description) 포함

### Step 3: 차시 구조 설계 (Backward Design Stage 3)

| 항목 | 내용 |
|------|------|
| 입력 | Step 0~2 결과, brainstorm_result.md §2(하위 주제), §5(관계 맵), §6(활동 아이디어), research_deep.md §3-2(확보된 소재) |
| 도구 | Read, Write |
| 산출물 | (Step 5에서 architecture.md에 통합) |

**동작**:

#### 3-1. 일별 테마 결정

나선형 커리큘럼 원칙에 따라 일별 테마를 설정한다:

```
[일별 테마 설계 원칙]

1. Bloom's 수준 점진적 상승:
   Day 1: 기억·이해 중심 (개념 소개, 환경 설정)
   Day 2: 이해·적용 중심 (기본 실습, 첫 적용)
   Day 3: 적용·분석 중심 (심화 실습, 패턴 분석)
   Day 4: 분석·평가·창조 (프로젝트 시작, 의사결정)
   Day 5: 창조·평가 중심 (프로젝트 완성, 발표, 회고)

2. 에너지 곡선 반영:
   오전: 신규 개념 도입 + 안내 실습 (I Do → We Do)
   오후: 독립 실습 + 프로젝트 (You Do)
   마지막 교시: 정리·복습·다음 날 예고

3. 일별 연결 구조:
   각 일의 산출물이 다음 날의 입력이 되는 누적 구조
   Day N 마무리: "오늘의 산출물" 확인
   Day N+1 시작: "어제의 산출물" 위에 쌓기
```

일별 테마 결정 시 brainstorm_result.md §5(관계 맵)의 선후/의존 관계를 반드시 반영한다.

#### 3-2. 하위 주제 → 차시 배정

brainstorm_result.md §2의 하위 주제를 일별 테마에 배정한다:

**배정 알고리즘**:

```
1. 선수 의존성 그래프 구성 (§5 관계 맵 기반)
   - 위상 정렬(topological sort)로 순서 결정
   - 선후 관계(→): 반드시 먼저 배치
   - 의존 관계(⇒): 같은 날 또는 이전 날 배치
   - 병렬 관계(||): 순서 무관, 시간 최적화로 배치

2. Bloom's 수준별 일 배정
   - 기억·이해 → Day 1~2
   - 적용 → Day 2~3
   - 분석 → Day 3~4
   - 평가·창조 → Day 4~5

3. 우선순위별 배정
   - Must: 반드시 배정 (시간 우선 확보)
   - Should: Must 배정 후 남은 시간에 배정
   - Could: 자율학습 자료로 전환 또는 시간 여유 시 배정

4. 일일 시간 검증
   - 각 일의 배정 시간 합 ≤ 일일 가용 시간
   - 차시당 신규 개념 ≤ 5개
   - 연속 이론 강의 ≤ 2교시 (3교시째부터 반드시 실습/활동)
```

#### 3-3. 차시별 상세 배치

각 차시(교시)의 내부 구조를 설계한다:

```
[1교시 (50분) 내부 구조]

도입 (5~8분, 10~15%):
  - 주의 집중 (Gagne #1): 흥미 유발 질문/사례/시연
  - 목표 안내 (Gagne #2): 이 교시에서 배울 것
  - 선행학습 회상 (Gagne #3): 전 교시/전일 내용 연결

전개 (35~40분, 70~80%):
  - 내용 제시 (Gagne #4): 개념 설명 (15-20분 세그먼트)
  - 학습 안내 (Gagne #5): 비유, 예시, 시연
  - 수행 유도 (Gagne #6): 실습, 연습
  - 피드백 제공 (Gagne #7): 즉각 확인, 교정

정리 (5~7분, 10~15%):
  - 수행 평가 (Gagne #8): 형성평가 (퀴즈/체크포인트)
  - 파지와 전이 (Gagne #9): 핵심 요약, 다음 교시 예고
```

이론-실습 비율은 `input_data.json`의 `pedagogy` 필드를 참조한다:
- "실습 비율 50% 이상"이면: 이론 20~25분 + 실습 20~25분 + 도입/정리 5~10분
- Gradual Release 적용: I Do(강사 시연) → We Do(함께 실습) → You Do(독립 실습)

#### 3-3.5. 일별 시간표(Timetable) 생성

차시별 상세 배치 완료 후, 일별 시간표 요약 테이블을 생성한다.

**기본값**:
- `start_time`: 09:00 (input_data.json에 별도 지정이 없으면)
- `lunch_duration`: 60분
- `lunch_after_session`: 4 (4교시 종료 후 점심)

**시간 계산 공식**:

```
교시 N 시작 시간 = start_time + (N-1) × (session_minutes + break_minutes)
교시 N 종료 시간 = 시작 시간 + session_minutes
점심 시간 = lunch_after_session번째 교시 종료 시간 ~ 종료 시간 + lunch_duration
점심 이후 교시 = 점심 종료 시간부터 재개

특수 교시:
- 아침 루틴 (0.5교시): session_minutes ÷ 2 = 25분, 이후 5분 전환
- Day 1 오리엔테이션: 정규 교시로 처리 (시간 계산 동일)
```

**출력 형식**: architecture.md §4-2.5에 Day별 시간표 테이블을 작성한다.

| 시간 | 구분 | 교시 | 내용 |
|------|------|------|------|
| 09:00~09:50 | 수업 | D{N}-1 | {하위 주제명} |
| 09:50~10:00 | 휴식 | - | - |
| ... | ... | ... | ... |
| 12:50~13:50 | 점심 | - | - |
| 13:50~14:40 | 수업 | D{N}-5 | {하위 주제명} |
| ... | ... | ... | ... |

#### 3-4. 시간 배분 검증

설계된 차시 구조가 현실적인지 검증한다:

| 검증 항목 | 기준 | 위반 시 조치 |
|-----------|------|------------|
| 일일 총 시간 ≤ 가용 시간 | 배정 시간 합 계산 | 초과 항목 시간 압축 또는 다음 날로 이동 |
| 차시당 신규 개념 ≤ 5개 | 해당 차시의 개념 수 카운트 | 차시 분할 또는 개념 통합 |
| 연속 이론 ≤ 2교시 | 연속 강의 교시 수 | 실습/활동 교시 삽입 |
| 실습 비율 ≥ 50% | 일별 실습 교시 / 총 교시 | 이론 교시를 실습으로 전환 |
| Must 커버리지 100% | Must 항목 미배정 여부 | 미배정 항목 강제 배정 |

검증 실패 항목이 있으면 Step 3-2로 돌아가 재배정한다 (최대 2회 반복).

### Step 4: 정렬 맵 작성 (Constructive Alignment)

| 항목 | 내용 |
|------|------|
| 입력 | Step 1~3 결과 |
| 도구 | Read, Write |

**동작**:

#### 4-1. 학습목표 ↔ 학습활동 ↔ 평가방법 매트릭스

3열 정렬 매트릭스를 작성한다:

```
| CLO | SLO | Bloom's | 하위 주제 | 학습활동 | 평가방법 | 차시 |
|-----|-----|---------|----------|---------|---------|------|
| CLO1 | SLO1-1 | 기억 | {주제} | {활동} | {평가} | D1-1 |
| CLO1 | SLO1-2 | 적용 | {주제} | {활동} | {평가} | D1-3 |
| ...  | ...    | ...  | ...    | ...    | ...    | ...  |
```

#### 4-2. 누락 검증

정렬 맵에서 다음 누락을 검출한다:

| 검증 | 조건 | 위반 시 |
|------|------|--------|
| **목표 커버리지** | 모든 CLO에 1개 이상 SLO가 매핑 | 누락 CLO에 SLO 추가 |
| **활동 커버리지** | 모든 SLO에 1개 이상 학습활동이 매핑 | 활동 추가 |
| **평가 커버리지** | 모든 SLO에 1개 이상 평가방법이 매핑 | 평가 추가 |
| **Bloom's 적합성** | 활동 유형이 Bloom's 수준에 적합 | 활동 유형 교체 |

위반 발견 시 해당 행을 수정하거나 추가한다.

#### 4-3. 학습 수준 진행 확인

정렬 맵의 차시 순서대로 Bloom's 수준이 점진적으로 상승하는지 확인:

```
Day 1: 기억·이해 ─────────┐
Day 2: 이해·적용 ──────────┤ 점진적 상승
Day 3: 적용·분석 ──────────┤
Day 4: 분석·평가·창조 ─────┤
Day 5: 창조·평가 ──────────┘

위반: Day 2에 '창조' 수준 활동이 배치되면 경고
     → 해당 활동을 Day 4 이후로 이동하거나, 사전 스캐폴딩 추가
```

#### 4-4. 콘텐츠 우선순위 매핑

정렬 맵에 우선순위 열을 추가하여 Must/Should/Could가 시간순으로 어떻게 분포되는지 시각화한다:

```
Day 1: [Must][Must][Must][Must][Must][Must][Should][Should]
Day 2: [Must][Must][Must][Should][Must][Must][Must][Should]
Day 3: [Must][Must][Must][Must][Should][Should][Must][Could]
Day 4: [Must][Must][Should][Must][Must][Must][Should][Could]
Day 5: [Must][Must][Must][Must][Should][Should][Should][Could]
```

Must가 전체의 70~80%를 차지하는지 확인한다.
